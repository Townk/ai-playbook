# CLI help, man pages, and zsh completion (Design)

**Status:** approved (2026-07-03) · infrastructure track

## Problem

The CLI's only help is a single 400-char `usage()` one-liner printed to stderr;
no subcommand has authored `--help` text (some internal ones leak Go's raw flag
list, `run`/`validate`/`env` suppress even that). There are no man pages and no
shell completion. Everything is hand-rolled `switch` + `flag.FlagSet` per
subcommand — no CLI framework.

## Goal

Comprehensive `--help`, generated man pages, and a zsh completion script — all
driven from **one command-metadata registry** so they never drift. Keep the
existing dispatch and per-command `flag.FlagSet` parsing intact (no cobra
migration).

## Design

### The registry — `internal/climeta` (new, importable)
One source of truth for the command surface, plus pure renderers:

```go
type Flag struct {
    Name        string // "--auto"
    Placeholder string // "<json|file>" (empty for bool flags)
    Desc        string // verbatim from the flag.FlagSet usage string
    Bool        bool
}
type Command struct {
    Name     string
    Aliases  []string // e.g. assist ⇐ troubleshoot
    Summary  string   // one line, for the overview
    Synopsis string   // "ai-playbook run [<slug> | --file <path>] [--assisted | --auto …]"
    Long     string   // paragraph(s) for per-command help + man DESCRIPTION
    Args     string   // "[<slug>]", "<prompt>", "<file.md>", …
    Flags    []Flag
    Examples []string // full command lines
    Internal bool     // grouped separately in --help; minimal man/completion
    SlugArg  bool     // takes a store <slug> positional ⇒ dynamic slug completion
}
var Commands []Command
```

Renderers (pure, string-returning, unit-testable):
- `Overview() string` — top-level `--help`: a short intro line, then commands
  grouped **user-facing** (assist, create, list, search, show, edit, run,
  validate, env) and **internal / advanced** (finalize, session, answer, mcp,
  input, diff, selftest, version) with aligned one-line summaries, and a
  `Run 'ai-playbook <command> --help' for details.` footer.
- `Help(name string) (string, bool)` — per-command help: synopsis, Long, a
  flags table (name+placeholder → desc), and examples. Resolves aliases.
- `Man(c Command) string` / `ManOverview() string` — roff (man(7)) for the
  overview page + one per command.
- `Zsh() string` — the `_ai-playbook` completion script.

**Scope of the registry data (from the verified inventory):** user-facing
commands are documented fully (every flag with its verbatim description, args,
constraints, 1–3 examples). Internal commands get a Summary + Synopsis + key
flags; **`input`'s ~40 theme flags are NOT enumerated** — it gets a one-line
"internal input widget" summary and its handful of behavioral flags only
(`--type`, `--out`, `--measure`), since it's launcher-invoked, not user-facing.

### `--help` wiring — `cmd/ai-playbook/main.go`
- `usage()` is replaced by `climeta.Overview()` (printed to **stdout** for
  `--help`/`-h`/`help`; stderr + exit 2 kept for the no-args / unknown-subcommand
  error paths).
- `help [<command>]` and a `-h`/`--help` token anywhere in a subcommand's args
  are intercepted **before** the subcommand parses: render `climeta.Help(cmd)` to
  stdout, exit 0. (A tiny `wantsHelp(args) bool` scans for `-h`/`--help`.) This
  gives every command real help without touching its `flag.FlagSet`.

### Man pages — generated, checked in, packaged
- `cmd/docgen` (new tool) imports `climeta` and writes `docs/man/ai-playbook.1`
  plus `docs/man/ai-playbook-<cmd>.1` for each non-internal command (and
  `finalize`, which is user-invokable). Roff hand-emitted from the registry (no
  new dependency) — `.TH`, `.SH NAME/SYNOPSIS/DESCRIPTION/OPTIONS/EXAMPLES`, a
  `SEE ALSO` cross-linking the pages.
- `//go:generate go run ./cmd/docgen` + a `make docs` target regenerate them; the
  results are committed so they're reviewable and always shippable.
- `.goreleaser.yml` archives gain `docs/man/*.1` (and the completion below) via
  the archive `files:` list, so release tarballs carry them.

### zsh completion — generated, dynamic slugs
- `climeta.Zsh()` emits `completions/_ai-playbook`: a `#compdef ai-playbook`
  script with a `_arguments`/`_describe` state machine — top-level subcommands
  (with summaries), then per-command flags. Generated by `cmd/docgen` alongside
  the man pages; committed.
- **Dynamic store-slug completion** for `run`/`show`/`edit`/`validate`/`env`
  (and `run --playbook`): a helper that runs `ai-playbook list --format
  fuzzy-data-source`, splits each line on `\x1f`, and offers field-2 (slug) as
  the candidate with field-1 (display) as the description.
- `--format` values (`human|fuzzy-data-source|json`) complete for `list`/`search`;
  `run` mode flags complete with their mutual-exclusion respected where practical.
- Package `completions/_ai-playbook` in the release archive; README documents
  install (`fpath` / the standard zsh site-functions location).

### Drift guard
A test asserts that for every **user-facing** command, each flag registered in
its `flag.FlagSet` (parsed by re-invoking `resolve*Args`-style definitions or by
a small exported flag-name list) has a matching `climeta` `Flag` entry — so a new
flag can't ship without help/man/completion coverage. (Internal commands, esp.
`input`, are exempt.)

## Non-goals
- No cobra/urfave migration; dispatch and `flag.FlagSet` parsing stay.
- No bash/fish completion (zsh only, per the project's shell); the registry makes
  adding them later trivial.
- Not documenting `input`'s full theme-flag surface.
- No Homebrew formula (deferred with the earlier release decision).

## Testing
- `climeta`: `Overview` lists every command once, grouped correctly; `Help`
  resolves aliases and includes each flag's description + examples; `Man` emits
  valid roff structure (`.TH`, the required `.SH` sections); `Zsh` includes each
  subcommand and the dynamic-slug hook.
- `main`: `ai-playbook --help` / `help run` / `run --help` print the registry
  help to stdout, exit 0; the drift-guard sync test.
- `docgen`: running it produces the expected files; the committed files match a
  fresh generation (a `make docs && git diff --exit-code docs/man completions`
  check, mirrored in a test or CI note).
